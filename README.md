# 温馨小镇（Cozy Town）

> 一个受“温馨生活模拟”设计方向启发、但完全原创内容的单机 2D 浏览器游戏原型。

## 1) 面向新手、空仓库的实现方案（先设计后编码）

### 目标边界
- **平台**：浏览器（单机、离线可运行）
- **画面**：2D
- **定位**：低学习成本、容易维护、可逐步迭代
- **合规约束**：不复刻任何现有游戏的角色、名称、地图、UI 或素材

### 基础工程原则
1. **小步快跑**：每个阶段只做“可运行 + 可验证”的最小功能。
2. **目录稳定**：尽早固定目录结构，后续按模块追加文件。
3. **零构建优先**：早期尽量不引入打包工具，降低调试复杂度。
4. **先功能后美术**：用占位图形/颜色块验证玩法循环。
5. **阶段验收**：每个阶段必须有“如何验证是否完成”。

---

## 2) 技术栈默认选择：单机 2D 浏览器游戏

当前仓库采用：
- `HTML + CSS + JavaScript + Canvas`
- 本地通过静态服务器运行（例如 Python `http.server`）

---

## 3) 两种方案对比

| 维度 | 原生 HTML/CSS/JS + Canvas | Phaser |
|---|---|---|
| 上手成本 | 最低，依赖少 | 中等，需要理解引擎生命周期与 API |
| 仓库初始复杂度 | 很低，几乎零配置 | 略高，需引入库和组织场景结构 |
| 可控性 | 极高，所有逻辑自己掌握 | 高，但受引擎设计约束 |
| 开发速度（小原型） | 快 | 快（但前期有学习成本） |
| 扩展到中型游戏 | 需要自建更多基础能力 | 更有优势（动画/相机/碰撞/资源管理） |
| 维护难度（新手） | 代码直观，但后期需自律架构 | 结构更规范，但要持续学习引擎 |
| 生态与文档 | 通用 Web 资料多 | 游戏方向文档更集中 |

---

## 4) 推荐方案与理由

### 推荐：**阶段 1~2 用原生 Canvas，阶段 3 后再评估是否迁移 Phaser**

为什么更适合“空仓库 + 单人开发 + 从零开始”：
1. **最少依赖，最快看到结果**：第一天就能跑出画面和循环。
2. **学习曲线平缓**：先建立游戏循环、输入、状态管理等核心认知。
3. **改动可控**：新手更容易理解每一行代码如何影响画面。
4. **可保留迁移选项**：当系统复杂到需要成熟引擎能力时，再迁移也不晚。

> 本仓库已先落地阶段1的最小可运行版本，便于后续逐阶段迭代。

## 5) 分阶段开发计划

## 阶段1：项目初始化与可运行原型（✅ 已实现）

- **目标**
  - 建立可运行的基础网页游戏骨架。
  - 有标题界面或基础画面，并具备最简单的状态切换。
- **需要新增的文件**
  - `index.html`
  - `styles.css`
  - `src/main.js`
  - `src/game.js`
  - `README.md`（本文件）
- **核心功能**
  - Canvas 渲染循环（`requestAnimationFrame`）。
  - 标题界面 + “开始界面占位”两种状态。
  - 键盘 Enter 切换状态。
- **如何验证是否完成**
  - 启动本地服务器，浏览器能打开页面。
  - 页面显示标题与画布。
  - 按 Enter 可在两种界面之间切换。

## 阶段2：角色移动与地图

- **目标**
  - 在地图上显示可控制角色，实现基础移动与碰撞边界。
- **需要新增的文件**
  - `src/systems/input.js`
  - `src/entities/player.js`
  - `src/world/tilemap.js`
  - `assets/maps/prototype-map.json`
- **核心功能**
  - WASD / 方向键移动。
  - 简单地块渲染（草地、道路、水域）。
  - 禁止角色走出地图边界。
- **如何验证是否完成**
  - 角色可平滑移动。
  - 到地图边缘停止，不会穿出画布。

## 阶段3：采集 / 背包 / 物品系统

- **目标**
  - 实现“靠近资源点并采集”的最小闭环。
- **需要新增的文件**
  - `src/entities/resource-node.js`
  - `src/systems/inventory.js`
  - `src/data/items.js`
  - `src/ui/inventory-panel.js`
- **核心功能**
  - 场景中生成可采集对象（例如木材、果实占位）。
  - 按键采集后进入背包。
  - 背包显示物品名称与数量。
- **如何验证是否完成**
  - 采集后资源消失或进入冷却。
  - 背包计数正确增加。

## 阶段4：时间系统与昼夜变化

- **目标**
  - 建立游戏内时间流逝与画面色调变化。
- **需要新增的文件**
  - `src/systems/game-time.js`
  - `src/rendering/day-night-overlay.js`
- **核心功能**
  - 游戏内分钟推进。
  - 清晨/白天/傍晚/夜晚不同色调。
  - 在 UI 上显示当前时间。
- **如何验证是否完成**
  - 放置一段时间后，时间与画面色调有明显变化。

## 阶段5：NPC 与简单对话

- **目标**
  - 加入可交互 NPC 与多句对话。
- **需要新增的文件**
  - `src/entities/npc.js`
  - `src/systems/dialogue.js`
  - `src/data/npc-dialogues.js`
  - `src/ui/dialogue-box.js`
- **核心功能**
  - 角色靠近 NPC 触发交互提示。
  - 按键打开对话框，逐句显示文本。
- **如何验证是否完成**
  - 可稳定触发对话，文本可翻页并关闭。

## 阶段6：存档与读取

- **目标**
  - 把核心进度写入本地存储并可恢复。
- **需要新增的文件**
  - `src/systems/save-load.js`
  - `src/ui/save-indicator.js`
- **核心功能**
  - 保存玩家位置、背包、游戏时间。
  - 启动时读取存档，支持新档/继续。
- **如何验证是否完成**
  - 刷新页面后数据能恢复。
  - 清档后回到初始状态。

## 阶段7：美化 UI 和音效占位

- **目标**
  - 在不改变核心玩法的前提下提升观感与反馈。
- **需要新增的文件**
  - `src/ui/theme.css`（或并入 `styles.css`）
  - `src/audio/audio-manager.js`
  - `assets/audio/`（占位音频）
  - `assets/ui/`（按钮/图标占位）
- **核心功能**
  - 统一 UI 配色与排版。
  - 按钮悬停/点击反馈。
  - 基础环境音和交互音效占位。
- **如何验证是否完成**
  - UI 风格统一、可读性提升。
  - 关键操作有可感知的音画反馈。

## 7) 如何运行（阶段1）

> 需本机已安装 Python 3（仅用于本地静态服务器）。

```bash
python3 -m http.server 8000
```

然后打开：

```text
http://localhost:8000
```

你会看到：
- 一个标题画面（Canvas 绘制）
- 按 `Enter` 可切换到“开始界面占位”
